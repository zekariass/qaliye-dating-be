# SuperLike Endpoint Description

## Endpoint
- **Method:** `POST`
- **Path:** `/api/v1/discovery/actions/superlike`
- **Authentication:** Bearer JWT (required)
- **Request Body:** JSON (required)

## Request
The request body must include both `targetUserId` and `clientActionId`:

```http
POST /api/v1/discovery/actions/superlike
Authorization: Bearer <jwt-token>
Content-Type: application/json

{
  "targetUserId": "uuid-of-target-user",
  "clientActionId": "uuid-generated-by-client"
}
```

### Request Fields
- **targetUserId** (UUID, required): The ID of the profile being superliked
- **clientActionId** (UUID, required): A unique ID generated by the client for idempotency. If the same `clientActionId` is sent multiple times, the server will return the same response without creating duplicate actions.

## Response
Returns a `SwipeActionResponse` object with the following structure:

```json
{
  "actionId": "uuid-of-created-action",
  "actionType": "SUPERLIKE",
  "status": "PENDING|MATCHED",
  "isMatch": true,
  "match": {
    "matchId": "uuid-of-match",
    "matchedAt": "2026-06-25T10:00:00Z",
    "rewindEligibleUntil": "2026-06-25T10:30:00Z",
    "otherUser": {
      "userId": "uuid-of-matched-user",
      "displayName": "Jane Doe",
      "primaryPhotoUrl": "https://..."
    }
  },
  "dailyLikesRemaining": 45,
  "dailySuperLikesRemaining": 0,
  "superLikeCreditsRemaining": 2,
  "createdAt": "2026-06-25T10:00:00Z",
  "idempotent": false
}
```

## Field Descriptions

### Action Details
- **actionId** (UUID): The ID of the created superlike action
- **actionType** (String): Always "SUPERLIKE" for this endpoint
- **status** (String): "PENDING" if no match yet, "MATCHED" if it resulted in a match
- **createdAt** (Instant): Timestamp when the action was created
- **idempotent** (Boolean): True if this was a duplicate request (same clientActionId already processed), false if this is a new action

### Match Information
- **isMatch** (Boolean): Whether the superlike resulted in an immediate match
- **match** (MatchSummaryDto | null): Match details if a match occurred, null otherwise
  - **matchId** (UUID): ID of the match
  - **matchedAt** (Instant): When the match was created
  - **rewindEligibleUntil** (Instant): Deadline until which this match can be rewound
  - **otherUser** (MatchedUserSummaryDto): Summary of the matched user
    - **userId** (UUID): ID of the matched user
    - **displayName** (String): Display name of the matched user
    - **primaryPhotoUrl** (String): URL of their primary photo

### Limits
- **dailyLikesRemaining** (Integer | null): Number of regular likes remaining today (null if unlimited)
- **dailySuperLikesRemaining** (Integer | null): Number of superlikes remaining today (null if unlimited)
- **superLikeCreditsRemaining** (Integer): Number of superlike credits available to use when daily limit is exceeded

## Frontend Implementation Notes

1. **Generate clientActionId:** Before making the request, generate a unique UUID for `clientActionId`. This ensures idempotency - if the request fails and is retried with the same ID, no duplicate action will be created.

2. **Handle match result:** If `isMatch` is true, show a match screen/animation with the matched user's details from `match.otherUser`.

3. **Display remaining counts:** Update UI to show `dailySuperLikesRemaining` and `superLikeCreditsRemaining` so users know their limits.

4. **Idempotent responses:** If `idempotent` is true, this was a retry of a previous request. The response contains the original action details - handle this gracefully (don't show duplicate notifications).

5. **Show rewind eligibility:** If a match occurs, you can show users that they can rewind within the `rewindEligibleUntil` timeframe.

## Error Responses

### 400 Bad Request - Validation Failed
```json
{
  "errorCode": "VALIDATION_ERROR",
  "message": "Validation failed",
  "errors": [
    {
      "field": "targetUserId",
      "message": "must not be null"
    },
    {
      "field": "clientActionId",
      "message": "must not be null"
    }
  ]
}
```
Required fields are missing or null in the request body.

### 400 Bad Request - Self Action
```json
{
  "errorCode": "SELF_ACTION_NOT_ALLOWED",
  "message": "You cannot perform actions on yourself."
}
```
User tried to superlike their own profile.

### 403 Forbidden - Account Ineligible
```json
{
  "errorCode": "ACCOUNT_INELIGIBLE",
  "message": "Your account is not eligible for discovery features."
}
```
User's account is not eligible for discovery (e.g., banned, suspended).

### 403 Forbidden - Profile Incomplete
```json
{
  "errorCode": "PROFILE_INCOMPLETE",
  "message": "Please complete your profile to use discovery features."
}
```
User's profile is incomplete and needs to be filled out.

### 429 Too Many Requests
```json
{
  "errorCode": "DAILY_SUPERLIKE_LIMIT_EXCEEDED",
  "message": "You have reached your daily super-like limit."
}
```
User has exceeded their daily superlike limit and has no superlike credits remaining.

## Similar Endpoints

The like and pass endpoints have the same request/response structure:

- **Like:** `POST /api/v1/discovery/actions/like` (same request body, `actionType` will be "LIKE")
- **Pass:** `POST /api/v1/discovery/actions/pass` (same request body, `actionType` will be "PASS")

Pass actions will never result in a match (`isMatch` will always be false, `match` will be null).
